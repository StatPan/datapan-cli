# Registry Release Artifacts

This document defines the first Datapan registry release surface. The goal is
to keep the `datapan-registry` repository boring to operate: it should publish
known artifacts produced by the CLI, not invent a second source of truth.
Schema naming and compatibility rules are defined in `docs/spec-governance.md`.

The release surface starts small:

- normalized specs;
- provider index;
- catalog audit;
- dependency inventory;
- adapter targets;
- route disposition;
- provider backlog;
- claim coverage report;
- verification evidence;
- schemas for those artifacts;
- a schema index for release consumers;
- provenance notes that explain how the files were produced.

Catalog audit is both a release artifact and a release gate. The artifact makes
catalog coverage gaps reviewable without requiring consumers to rerun the audit
locally.

`datapan-registry` releases use this layout:


copied into the registry repository.

## Artifact Contracts

### Normalized Specs

Path:

```text
data/data-go-kr.registry.json
```

Schema:

```text
schemas/datapan.specs.v1.schema.json
```

Generated by:

```bash
datapan catalog update data-go-kr --registry data/data-go-kr.registry.json --apply --backup --diff-limit 0 --json
```

Installed by consumers with:

```bash
datapan catalog install datapan-registry --registry .datapan/data-go-kr.registry.json --json
```

The specs registry preserves upstream data.go.kr catalog metadata and keeps
Datapan-created search helpers separate from upstream fields.

### Registry Shards

Path:

```text
registry-shards.json
```

Schema:

```text
schemas/datapan.registry-shards.v1.schema.json
```

Registry shards are optional consumer acceleration artifacts, normally
published outside the main release zip as `data-go-kr-shards.tar.gz`. Consumers
must keep `data/data-go-kr.registry.json` as the compatibility fallback and
validate shard inventories against the canonical registry checksum before using
shard files.

### Provider Index

Path:

```text
data/provider-index.json
```

Schema:

```text
schemas/datapan.provider-index.v1.schema.json
```

Generated by:

```bash
datapan catalog release draft --registry data/data-go-kr.registry.json --output-dir .datapan/release --json
```

The provider index lists provider adapters registered in the Datapan runtime,
their owned hosts, declared capabilities, and `split_readiness` for deciding
whether the adapter boundary is ready to move into a separate
`datapan-providers` package. The provider backlog says which host families
still need work; the provider index says which host families Datapan currently
claims as adapter-owned.

### Coverage Report

Path:

```text
reports/coverage.json
```

Schema:

```text
schemas/datapan.coverage.v1.schema.json
```

Generated by:

```bash
datapan catalog coverage --registry data/data-go-kr.registry.json --verification reports/latest-verification.json --route-disposition reports/route-disposition.json --output reports/coverage.json --json
```

The coverage report is the release's claim-level dashboard. It records callable
operation coverage, external adapter coverage, verification evidence coverage,
provider split readiness, and top adapter gaps so maintainers can see whether
Datapan is moving toward broader public-data coverage or merely accumulating
files.

### Runtime Evidence Growth Schema

Path:

```text
schemas/datapan.runtime-evidence-growth.v1.schema.json
```

The runtime evidence growth schema lets registry releases describe source-scoped
verification coverage against an explicit target. The schema is part of the
release schema inventory and validates
`reports/data-go-kr/runtime-evidence-growth.json`, which records the current
runtime evidence percentage, target evidence count, remaining gap, next planned
verification batches, provider split readiness, and any below-target warning.

### Studio Schemas

Paths:

```text
schemas/datapan.studio-datasets.v1.schema.json
schemas/datapan.studio-bundle.v1.schema.json
```

The Studio schemas define the static consumer bundle generated by
`datapan catalog studio`. They let a future UI or local viewer consume dataset
cards, operation summaries, starter-kit examples, provider readiness, and
overview counts without inventing a separate metadata model. The generated
bundle also includes `index.html`, a static local viewer over the same bundle
data.

### Catalog Audit

Path:

```text
reports/catalog-audit.json
```

Schema:

```text
schemas/datapan.catalog-audit.v1.schema.json
```

Generated by:

```bash
datapan catalog audit --registry data/data-go-kr.registry.json --output reports/catalog-audit.json --json
datapan catalog coverage --registry data/data-go-kr.registry.json --verification reports/latest-verification.json --route-disposition reports/route-disposition.json --json
```

The catalog audit records registry coverage, missing metadata, callable
operation counts, dependency-class counts, top endpoint/guide hosts, and bounded
samples for the biggest gaps. It is the release artifact that proves Datapan is
being honest about what is known, callable, external, malformed, approval-gated,
or unsupported.
Use `catalog coverage` after audit and verification to review the release's
claim-level coverage: callable operations, external adapter coverage,
verification evidence coverage, provider split readiness, and the largest
remaining adapter gaps. When `reports/route-disposition.json` is available,
coverage also includes evidence-adjusted missing route counts so stale or
transient upstream routes do not look like ordinary adapter backlog.

### Catalog Diff

Path:

```text
reports/catalog-diff.json
```

Schema:

```text
schemas/datapan.catalog-diff.v1.schema.json
```

Generated by:

```bash
datapan catalog diff --old previous/data-go-kr.registry.json --new data/data-go-kr.registry.json --limit 0 --output reports/catalog-diff.json --json
```

The catalog diff records added, removed, and changed datasets by stable list ID.
It is optional when a previous registry is not available, but releases should
include it whenever `--previous-registry` is provided so automation can produce
consistent change logs before a registry snapshot is published.

### Error Catalog

Path:

```text
reports/error-catalog.json
```

Schema:

```text
schemas/datapan.error-catalog.v1.schema.json
```

Generated by:

```bash
datapan catalog errors --registry data/data-go-kr.registry.json --limit 0 --output reports/error-catalog.json --json
```

The error catalog inventories provider status fields declared by upstream
response parameters. It preserves names such as `resultCode`, `resultMsg`,
`returnReasonCode`, `returnAuthMsg`, and `errMsg`, then adds conservative
Datapan roles so verification, SDK, and Studio layers can keep provider errors
visible without inventing a parallel taxonomy.

### Dependency Inventory

Path:

```text
reports/dependencies.json
```

Schema:

```text
schemas/datapan.dependencies.v1.schema.json
```

Generated by:

```bash
datapan catalog dependencies --registry data/data-go-kr.registry.json --limit 0 --output reports/dependencies.json --json
```

The dependency inventory records operation-level dependency classification. It
keeps exact dataset IDs, operation names, endpoint/source/guide hosts,
dependency classes, adapter status, provider families, approval metadata, skip
reasons, and parameter counts together. The provider backlog summarizes this
evidence by host; the dependency inventory keeps the per-operation evidence
that agents, SDK generators, and a future Studio need before deciding whether
an operation can be called or needs adapter work.

### Adapter Targets

Path:

```text
reports/adapter-targets.json
```

Schema:

```text
schemas/datapan.adapter-targets.v1.schema.json
```

Generated by:

```bash
datapan catalog adapter-targets --registry data/data-go-kr.registry.json --limit 0 --output reports/adapter-targets.json --json
```

Adapter targets turn missing external endpoint and service-root operations into
a ranked work queue by host. They keep provider family, dependency kinds,
operation and spec coverage, organizations, formats, approval and missing
parameter counts, unsupported protocol counts, and bounded sample operations in
one artifact. This is the release artifact to use when deciding which external
provider adapter should be implemented next.

### Route Disposition

Path:

```text
reports/route-disposition.json
```

Schema:

```text
schemas/datapan.route-disposition.v1.schema.json
```

Generated by:

```bash
datapan catalog route-disposition --registry data/data-go-kr.registry.json --probe reports/unadapted-external-probe.json --limit 0 --output reports/route-disposition.json --json
```

Route disposition combines missing external endpoint operations with
credential-free unadapted probe evidence. It classifies each missing route as a
dead-route candidate, transient failure, parameter-blocked route, or remaining
adapter candidate so adapter work starts from evidence rather than a raw host
count.

### Provider Backlog

Path:

```text
reports/provider-backlog.json
```

Schema:

```text
schemas/datapan.providers.v1.schema.json
```

Generated by:

```bash
datapan catalog providers --registry data/data-go-kr.registry.json --limit 0 --output reports/provider-backlog.json --json
```

The provider backlog explains external host/provider gaps. It is the first
input for deciding which provider adapter should be built next.

### Verification Evidence

Path:

```text
reports/latest-verification.json
reports/verification-plan.json
reports/latest-verification-summary.json
```

Schema:

```text
schemas/datapan.verification.v1.schema.json
schemas/datapan.verification-plan.v1.schema.json
schemas/datapan.verification-summary.v1.schema.json
```

Generated by:

```bash
datapan catalog verify --registry data/data-go-kr.registry.json --limit 100 --timeout 10s --output reports/latest-verification.json --json
datapan catalog verify plan --registry data/data-go-kr.registry.json --verification reports/latest-verification.json --output reports/verification-plan.json --json
datapan catalog verify summary --input reports/latest-verification.json --output reports/latest-verification-summary.json --json
```

The verification report is evidence, not a broad guarantee. It records
`verified`, `failed`, `skipped`, and `unknown` status conservatively. Skips are
expected when an API needs an external adapter, usage approval, required
parameters, or an unsupported protocol.
The verification summary groups the evidence by status, reason, provider,
endpoint host, and dependency class so release reviewers can see coverage gaps
without scanning every operation.
The verification plan is the cadence helper: it reads current evidence and
prints the next `catalog verify` batches with `--exclude-input`, so maintainers
can increase evidence coverage without rechecking operations already represented
in the release.

### Schema Index

Path:

```text
schemas/index.json
```

Schema:

```text
schemas/datapan.schema-index.v1.schema.json
```

Generated by:

```bash
datapan catalog release draft --registry data/data-go-kr.registry.json --output-dir .datapan/release --json
```

The schema index lists every Datapan schema shipped in the release with its
relative path, `$id`, title, contract name, version, byte size, and SHA-256
checksum. It gives future SDK generators, Studio, registry consumers, and
release automation one stable discovery point instead of requiring them to
guess which schema files exist in a release.

### Release Manifest

Path:

```text
manifest.json
```

Schema:

```text
schemas/datapan.release-manifest.v1.schema.json
schemas/datapan.release-verification.v1.schema.json
schemas/datapan.release-readiness.v1.schema.json
```

Generated by:

```bash
datapan catalog release draft --registry data/data-go-kr.registry.json --previous-registry previous/data-go-kr.registry.json --verification reports/latest-verification.json --output-dir .datapan/release --json
```

The manifest records release artifact paths relative to the release root, byte
sizes, SHA-256 checksums, artifact kinds, and schema IDs when an artifact has a
Datapan schema. The manifest does not include itself, so its own checksum does
not make the release unstable.

`catalog release verify --manifest PATH --output REPORT --json` emits and
stores a `datapan.release-verification.v1` report that records the checked
artifact count, failure count, and per-artifact verification status. It also
validates schema-bound artifacts against the schema files shipped in the same
release, including the manifest itself when the release manifest schema is
available under `schemas/`.

`catalog release readiness --manifest PATH --output REPORT --json` emits and
stores a `datapan.release-readiness.v1` report. It is the publication gate for
deciding whether the draft is ready to become a `datapan-registry` release:
manifest verification, required artifacts, complete schema set, and non-empty
registry are hard gates; catalog diff and runtime verification artifacts are
recommended gates that remain visible as warnings.

## Local Release Draft

A local draft can be prepared under `.datapan/release/` before publishing to
`datapan-registry`:

```bash
datapan catalog update data-go-kr --registry .datapan/data-go-kr.registry.json --enrich-link-details --backup --apply --json
datapan catalog release draft --registry .datapan/data-go-kr.registry.json --previous-registry .datapan/previous.registry.json --verification .datapan/latest-verification.json --output-dir .datapan/release --provider-limit 0 --json
datapan catalog diff --old .datapan/previous.registry.json --new .datapan/release/data/data-go-kr.registry.json --limit 0 --output .datapan/release/reports/catalog-diff.json --json
datapan catalog audit --registry .datapan/release/data/data-go-kr.registry.json --json
datapan catalog dependencies --registry .datapan/release/data/data-go-kr.registry.json --limit 0 --output .datapan/release/reports/dependencies.json --json
datapan catalog adapter-targets --registry .datapan/release/data/data-go-kr.registry.json --limit 0 --output .datapan/release/reports/adapter-targets.json --json
datapan catalog route-disposition --registry .datapan/release/data/data-go-kr.registry.json --probe .datapan/release/reports/unadapted-external-probe.json --limit 0 --output .datapan/release/reports/route-disposition.json --json
datapan catalog verify --input .datapan/release/reports/latest-verification.json --status failed --json
datapan catalog verify plan --registry .datapan/release/data/data-go-kr.registry.json --verification .datapan/release/reports/latest-verification.json --output .datapan/release/reports/verification-plan.json --json
datapan catalog verify summary --input .datapan/release/reports/latest-verification.json --json
datapan catalog release verify --manifest .datapan/release/manifest.json --output .datapan/release/reports/latest-release-verification.json --json
datapan catalog release readiness --manifest .datapan/release/manifest.json --output .datapan/release/reports/latest-release-readiness.json --json
```

When `--verification` is provided, `catalog release draft` writes both
`reports/latest-verification.json` and
`reports/latest-verification-summary.json`.
When `reports/unadapted-external-probe.json` already exists in the release
output directory, the draft normalizes it, regenerates
`reports/unadapted-external-probe-summary.json`, and includes both as
schema-bound manifest artifacts.
Every draft also writes `data/provider-index.json`,
`reports/catalog-audit.json`, `reports/error-catalog.json`,
`reports/dependencies.json`, `reports/adapter-targets.json`,
`reports/route-disposition.json`,
`reports/provider-backlog.json`,
`reports/coverage.json`, `reports/verification-plan.json`,
`schemas/index.json`, `RELEASE_NOTES.md`, and `manifest.json` with artifact
paths, byte sizes, and SHA-256 checksums. When `--previous-registry` is
provided, it also writes and manifests `reports/catalog-diff.json`.

The local draft directory is ignored by git in this repository. Published
registry artifacts now live in `https://github.com/StatPan/datapan-registry`.
The initial release is `v2026.06.24`, with the large registry JSON stored by
Git LFS and also attached as a zip release asset for non-LFS consumers.

`catalog release draft` does not fetch the upstream catalog and does not call
provider APIs. It assembles release artifacts from existing local inputs:
registry JSON, optional previous-registry diff, provider index, schema files,
catalog audit, error catalog, dependency inventory, adapter targets, route
disposition, and provider backlog generated from that registry, optional
verification evidence, optional unadapted external probe evidence, provenance notes, human-facing
release notes, and a manifest.

## Release Gates

Before publishing a registry snapshot:

- the specs file must be generated by `catalog update --apply`;
- `data/provider-index.json` must be generated from the registered provider
  adapters;
- `catalog audit` must run against the generated specs;
- `reports/catalog-audit.json` must be generated from the released registry;
- `reports/dependencies.json` must be generated with `--limit 0`;
- `reports/adapter-targets.json` must be generated with `--limit 0`;
- `reports/route-disposition.json` must be generated with `--limit 0`;
- when a previous registry exists, `reports/catalog-diff.json` must be generated
  with `--limit 0`;
- provider backlog must be generated with `--limit 0`;
- verification must be bounded and must state its `--limit`;
- verification reports must be readable again with `catalog verify --input`;
- verification summaries should be generated for provider/reason review;
- when coverage still reports missing adapter operations,
  `reports/unadapted-external-probe.json` and
  `reports/unadapted-external-probe-summary.json` must be present and
  manifest-bound;
- `schemas/index.json` must list every copied schema with size and SHA-256
  checksum;
- `manifest.json` must list every release artifact except itself with size and
  SHA-256 checksum;
- `RELEASE_NOTES.md` must be generated from the same release evidence and
  included in the manifest as `release_notes`;
- `catalog release verify --manifest manifest.json --output reports/latest-release-verification.json`
  must pass checksum and schema-bound artifact validation before publishing;
- `catalog release readiness --manifest manifest.json --output reports/latest-release-readiness.json`
  must pass required publication gates before publishing;
- credentials must never appear in artifacts;
- schema files must be copied from this repository, not rewritten by hand;
- provenance must record CLI version, source provider, generation time, and
  the exact commands used.

## Split Trigger

Keep publishing to `datapan-registry` only when all of these remain true:

- schemas are stable enough to publish;
- provider backlog can be generated repeatedly;
- verification reports are useful even when mostly skipped;
- release cadence is worth separating from CLI code;
- consumers can benefit from downloading artifacts without re-importing the
  upstream catalog themselves.

If those conditions stop holding, keep iterating in `datapan-cli` and treat
local `.datapan` files as draft artifacts until the release is trustworthy
again.
